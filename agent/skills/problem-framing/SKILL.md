# Problem Framing

## Purpose

Use this skill to turn fuzzy requests, messy systems, or overloaded choices into a clearer
problem statement, better boundaries, and a credible next move.

This skill is for orientation and judgment. It should reduce confusion, expose hidden risks,
and stop the system from sprinting in the wrong direction just because action feels easier
than thinking.

## When to Use

- The task is vague or cross-cutting
- The user is overwhelmed by too many options
- A system feels structurally wrong but the exact issue is unclear
- A decision needs tradeoff framing before execution
- An issue was discovered and the right corrective move is not obvious
- An agent, skill, or workflow needs recursive improvement guidance

## Rules

- Find the real objective before proposing work.
- Separate the underlying problem from the user’s first suggested solution.
- Distinguish uncertainty, risk, and missing information from actual blockers.
- Prefer a small number of strong options over a giant menu of possibilities.
- State tradeoffs directly: speed, risk, churn, complexity, ownership.
- If the structure is drifting, call that out instead of working around it silently.
- End with the next best move, not just analysis.

## Workflow

1. Clarify the objective.
   - What outcome matters, and what pain is the user actually trying to remove?
2. Identify the real shape of the problem.
   - Is this strategy, architecture, implementation, review, or recovery?
3. Surface constraints and risks.
   - Time, complexity, ownership, compatibility, confidence, and likely failure modes.
4. Reduce the decision set.
   - Present the strongest framing and the smallest useful set of options.
5. Recommend the next move.
   - What should happen now, and what can wait?

## Expected Output

- a sharper statement of the real problem
- the main risks, assumptions, and tradeoffs
- the smallest useful option set
- a concrete next step or owning lane
